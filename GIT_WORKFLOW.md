# Как работать с git без казусов

Короткая памятка для команды. Цель — чтобы ничьи изменения не терялись и не было конфликтов.

## Главное правило

**`git pull --rebase` перед началом работы И перед каждым пушем.**

git не затирает чужой код молча: разные файлы (или разные места одного файла) он сольёт сам, а одинаковые строки — остановится и попросит выбрать. Потерять чужую работу можно, только если делать `git push --force` (НЕ делаем, см. ниже).

## Цикл работы (делай так каждый раз)

```bash
# 1. перед работой — взять свежак
git pull --rebase

# 2. ... пишешь код ...

# 3. сохранить локально
git add <свои файлы>          # или: git add -A
git commit -m "что сделал"

# 4. перед отправкой — снова подтянуть чужое
git pull --rebase

# 5. отправить
git push
```

## Если `git push` отклонён (`! [rejected] ... fetch first`)

Это НЕ ошибка — просто кто-то запушил, пока ты работал. Делай:

```bash
git pull --rebase
git push
```

## Если возник конфликт

git напишет, в каком файле, и вставит туда маркеры:

```
<<<<<<< HEAD
твой вариант
=======
вариант тиммейта
>>>>>>> ...
```

Оставь нужное (или объедини оба), убери строки с `<<<<<<<`, `=======`, `>>>>>>>`, потом:

```bash
git add <файл>
git rebase --continue
git push
```

Если запутался — пиши в чат до того, как что-то делать. Не паникуй: пока не запушил, всё чинится.

## Чтобы конфликтов почти не было — работаем по зонам

- **Data/Backend** → `internal/ingest`, `internal/metrics`, `internal/storage`, `internal/categorize`, `db/`, `testdata/`
- **Fullstack / Product** → `internal/api`, `internal/llm`, `web/`, промпты
- **DevOps** → `deploy/`, `Makefile`, CI, миграции

Пока не лезем в одни и те же файлы — git всё сольёт автоматически.

## Запрещено

- ❌ `git push --force` в `main` — единственный способ реально затереть чужое.
- ❌ Коммитить `.env` и любые ключи/пароли (он в `.gitignore` — не вынимай).
- ❌ Коммитить, если `go build ./...` падает — не ломай сборку остальным.

## Перед коммитом — быстрая проверка

```bash
go build ./...      # собирается?
go test ./...       # тесты зелёные?
```

Если зелено — коммить и пушь.

## Совсем надёжно (опционально): ветки + PR

Вместо прямой работы в `main`:

```bash
git switch -c feat/мой-слой   # своя ветка
# ... работа, коммиты ...
git push -u origin feat/мой-слой
```

Потом открываешь Pull Request в `main` на GitHub → ревью → мёрж. Так `main` всегда рабочий.
